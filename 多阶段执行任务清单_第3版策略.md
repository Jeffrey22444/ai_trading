# Codex 多阶段执行任务清单：交易策略 v2 安全修复

## 背景

项目：AI trading / Hyperliquid 测试网交易系统。

问题：在 `9a91e43 feat: add strategy v2 quant guardrails` 之后，测试网交易结果出现 win rate 为零，且多笔交易方向与后续价格走势相反。

初步判断：当前问题不应按“LONG/SHORT 简单反转”处理。更合理的判断是：

1. 趋势追随评分过强；
2. 入场追价过滤不足；
3. 评分参考价、止损价、执行价不一致；
4. 缺少执行前价格漂移保护；
5. 交易复盘字段不足，无法清楚定位错误来源。

本任务目标是做保守止血版修复：**宁愿少交易，也不要继续追涨杀跌。**

# 阶段 4：执行前增加价格漂移和追价保护

## 目标

即使 AI 和 guardrail 都允许开仓，只要执行价相对评分参考价偏移过大，也不下单。

## 修改文件

```text
backend/trading/risk_guard.py
backend/agent/nodes/trading_execution_node.py
backend/tests/test_risk_guard.py
```

如已有 execution node 测试，则扩展；没有则新增最小测试。

## 修改 `risk_guard.py`

扩展：

```python
validate_open_decision()
```

新增参数：

```python
reference_price: float | None = None
max_entry_price_drift_pct: float | None = None
max_chase_price_drift_pct: float | None = None
```

## 规则

### 绝对漂移保护

如果 `reference_price` 存在，且 `max_entry_price_drift_pct` 存在：

```python
abs(current_price - reference_price) / reference_price > max_entry_price_drift_pct
```

则：

```python
raise ValueError
```

### 追价保护

如果 `max_chase_price_drift_pct` 存在：

```text
OPEN_LONG 且 current_price > reference_price * (1 + max_chase_price_drift_pct) → raise ValueError
OPEN_SHORT 且 current_price < reference_price * (1 - max_chase_price_drift_pct) → raise ValueError
```

### 异常 reference price

如果：

```python
reference_price <= 0
```

则：

```python
raise ValueError
```

### 保持原有逻辑

现有以下校验必须保持不变：

1. 实盘开关；
2. 仓位上限；
3. 止损止盈方向；
4. 仓位金额合法性。

## 修改 `trading_execution_node.py`

1. 从 `decision["quant_guardrail"]` 或 decision 中拿：

```python
reference_price
```

2. 调用 `validate_open_decision` 时传入：

```python
reference_price
config.execution_safety.max_entry_price_drift_pct
config.execution_safety.max_chase_price_drift_pct
```

3. 如果校验失败：

   - 不调用 `trader.open_long/open_short`；
   - `execution_status` 标记为 `failed` 或 `blocked`，保持现有状态语义一致；
   - `execution_result` 写入：

```python
error
reject_reason
reference_price
current_price
drift_pct
```

4. 不要吞掉错误导致看不见原因。

## 测试

在 `test_risk_guard.py` 中新增：

1. `OPEN_LONG` current_price 比 reference_price 高超过 `max_chase_price_drift_pct` 时拒绝。
2. `OPEN_SHORT` current_price 比 reference_price 低超过 `max_chase_price_drift_pct` 时拒绝。
3. current_price 相对 reference_price 绝对偏离超过 `max_entry_price_drift_pct` 时拒绝。
4. 偏离在阈值内时仍通过。
5. `reference_price <= 0` 时拒绝。
6. 原有 live trading 禁用、止损止盈方向测试仍通过。

如果新增 execution node 测试：


```python

```

2. 构造一个 `OPEN_SHORT` decision：

```text

```

```python

```

# 阶段 5：强化 AI Prompt 和策略文档

## 目标

让 AI 不再机械接受系统方向，而是明确承担“拒绝追价/拒绝趋势末端”的复核职责。

## 修改文件

```text
backend/config/trading_strategy.md
backend/agent/nodes/analysis_node.py
```

如果 `analysis_node.py` 已经足够清楚，只做必要最小修改。

## 修改 `trading_strategy.md`

加入以下原则：

1. `direction_bias` 只是候选方向，不是开仓理由。
2. 即使系统 `action_allowed=true`，AI 仍必须检查：

```text
是否追高/追低
是否 RSI 过热/过冷
是否 MACD 动能衰减
是否价格远离 EMA20
是否数据过期
是否无法解释入场优势
```

3. 满足上述任一风险时，AI 必须 HOLD。
4. AI 不得因为“系统允许开仓”就默认开仓。
5. 最高纪律：无法解释入场优势时 HOLD。

## 修改 `analysis_node.py`

确保 prompt 中明确写入：

```text
AI 可以把系统允许的 OPEN_LONG/OPEN_SHORT 改为 HOLD
AI 的主要职责是拒绝低质量入场
action_allowed=true 不是必须开仓
不得反向开仓
```

确保 `quant_guardrail` prompt dict 中包含：

```text
entry_quality
reference_price
```

## 测试

如果已有 `strategy_contract` 测试，运行相关测试。

否则至少运行：

```bash
cd backend
python -m pytest -q tests/test_quant_guardrails.py
```

## 成功标准

1. prompt 明确“允许开仓 ≠ 必须开仓”。
2. prompt 明确“追高/追空/趋势末端必须 HOLD”。
3. 不改变 JSON 输出 schema，除非测试同步更新。
